# 1. 设计目标与最终结论

系统目标不是“定时调用接口后直接写表”，而是建立一条可追溯、可重放、能识别半成品的数据生产链。任一正式数据都必须回答：由哪个采集批次产生、使用了哪些原始资产、经过哪个加工版本、何时发布、当前是否完整。

| 设计项 | 最终方案 | 原因 |
|-|-|-|
| 业务范围 | 沿用25张业务表和6张系统运行表 | 不新增实时、分钟、财务、周线和月线数据 |
| 数据库 | 单实例PostgreSQL 16 | 同时承担正式数据、运行状态、持久任务队列和APScheduler JobStore |
| 原始数据 | 保留Parquet，使用Zstandard压缩并不可变封存 | 支持低成本留存、批量读取、重放加工和供应方字段追溯 |
| 采集并发 | 默认最多4个采集任务并发，共享480次/分钟应用预算 | 提高采集效率；480为项目默认预算，各接口仍按账户权限配置更低上限 |
| 加工并发 | 全系统固定为1 | 控制数据库和磁盘IO，保证确定性顺序 |
| 任务入口 | 所有自动任务、补采和人工重跑统一写入数据库队列 | Web和CLI都不得绕过队列直接请求Tushare或写正式表 |
| 监控 | 项目内置监控页面、规则引擎和结构化日志 | 同时覆盖进程、数据库、接口、任务、数据发布和磁盘容量 |
| 中间件 | 不引入Redis、Kafka、Celery或Airflow | 当前是单机部署，PostgreSQL队列和单调度进程足以满足一致性及恢复要求 |

**流程修正。**确认稿中“08:45采集当日stk_limit”“09:25采集当日adj_factor”与“16:10创建DAILY批次”存在先后矛盾。实现时应在交易日08:45创建或复用当日DAILY批次并加入stk_limit，09:25向同一批次加入adj_factor；16:10只追加盘后第一阶段任务，不能再创建第二个DAILY批次。

**数据库范围。**本方案不新增表，也不把股票、ETF和指数合并。为使状态机和多资产依赖真正可执行，collection_batch补充计划冻结字段，processing_task补充自动重试字段，processing_dependency补充依赖类型、稳定范围键和已解析发布任务；同时补充相应幂等与重试索引。多用户操作审计表、分布式队列和多节点限流均不进入本期。

# 2. 总体架构与系统边界

```text
Vue管理端 ──HTTP──> FastAPI管理API ──读写运行状态──> PostgreSQL
                                      │
APScheduler计划器 ──创建批次/任务─────┤
                                      │
采集执行器 ──限流请求──> Tushare      │
    │                                 │
    └──封存──> Parquet原始资产 ────────┤
                                      │
串行加工执行器 ──读取资产/校验/转换────> 正式业务表
                                      │
                                      └──更新dataset_release

内置监控模块 ──查询运行表/数据库状态/本机资源/调度状态──> 管理端概览与告警中心
```

| 组件 | 职责 | 明确禁止 |
|-|-|-|
| Web管理端 | 展示批次、任务、依赖、队列、发布状态和告警；提交受控人工操作 | 不连接数据库，不直接调用Tushare |
| FastAPI | 提供查询模型、人工任务入口、健康检查和API指标 | 不执行长时间采集或加工 |
| Scheduler | 交易日门禁、批次规划、任务领取、采集执行、加工执行、恢复协调 | 不把进程内Future当作任务真相 |
| Tushare适配器 | SDK隔离、字段选择、平滑限流、超时、网络重试和请求指标 | 不包含业务表写入和跨接口关联 |
| 原始资产存储 | 保存采集任务封存后的Parquet文件 | 不保存已换算单位或加工后的字段 |
| PostgreSQL | 31张确认表、APScheduler JobStore、事务锁、运行队列和正式发布 | 不存放大块原始响应正文 |
| 监控组件 | 采集指标、规则告警、趋势看板和容量观测 | 不替代数据库中的精确运行记录 |

**控制面与数据面分离。**批次状态、任务状态、依赖和发布记录属于控制面，保存在PostgreSQL；Parquet和25张业务表属于数据面。内置监控只读取并聚合控制面状态、数据库状态和本机资源，不参与任务恢复；结构化日志用于定位单次尝试细节，不作为正式数据完整性的依据。

**四条系统不变量。**正式加工只能读取已封存资产；批次关闭后不能重新打开；同一时刻最多一个加工任务运行；消费者只能把dataset_release已发布的范围视为完整数据。

# 3. 部署拓扑与进程模型

生产环境采用单机原生进程部署，不使用Docker或其他容器。PostgreSQL、API、Scheduler和Web分别运行，由系统服务管理器或统一运行脚本维护生命周期；进程间通过本机回环地址或Unix Socket通信，数据库与管理端口不直接对外暴露。监控属于API、Scheduler和管理端自身功能，不启动独立监控服务。

| 服务 | 进程/监听 | 持久目录 | 健康判定 |
|-|-|-|-|
| postgres | PostgreSQL 16原生服务，本机5432或Unix Socket | data/postgres、logs/postgres | pg_isready和关键查询 |
| migrate | Alembic一次性命令 | 无 | 迁移退出码为0且版本一致 |
| server | Uvicorn/FastAPI，本机8000 | logs/server | /api/v1/health/live与/ready |
| scheduler | BlockingScheduler加内部工作线程，本机状态端口9101 | data/raw、logs/scheduler | 进程存活、调度心跳和最近轮询时间 |
| web | 静态资源服务与反向代理，作为唯一业务入口 | logs/web | HTTP健康页与API代理检查 |

Scheduler对data/raw拥有读写权限；API不直接读取原始目录，只通过数据库展示资产元数据。备份进程读取PostgreSQL备份目录和data/raw。数据库、原始资产和日志使用独立子目录，所有路径由配置显式给出，不依赖进程启动目录。

**线程模型。**API使用异步数据库会话；Scheduler使用同步会话。Scheduler内部建立一个最多4线程的采集执行器和一个固定1线程的加工执行器，两者不共用线程池，避免大量采集任务占满线程后饿死加工入口。APScheduler只负责触发计划、派发、恢复和维护作业，不承载业务任务的唯一状态。

**资源边界。**采集执行器最多4线程，加工执行器固定1线程；两者由队列深度、磁盘保留空间和数据库状态共同反压。历史回填以低优先级运行；出现积压时不通过增加加工并发解决。
