# 控制面运行模型

状态：已完成

## 目标与非目标

落地采集批次、采集任务、原始资产、加工任务、加工依赖和数据发布 6 张运行表，建立后续状态机和队列的数据库边界。本阶段不调用 Tushare、不写 Parquet、不实现业务表加工。

## 实现任务

1. 建立 `modules/acquisition` 与 `modules/processing`，删除过渡性的 `tasking` 模块和空路由。
2. 为全部状态、批次类型、依赖类型和发布范围建立统一枚举。
3. 实现 6 张 SQLAlchemy 模型及主键、外键、检查约束和幂等约束。
4. 建立活动批次、到期重试、加工队列、依赖解析和发布查询索引。
5. 编写首个 Alembic 迁移，支持完整升级与逆序回滚。
6. 增加元数据、约束和迁移 SQL 测试，验证 Alembic 只有一个 head。

## 验收条件

- ORM 元数据准确包含 6 张运行表，不再导入 `tasking`。
- PostgreSQL 离线迁移 SQL 能生成全部表、索引和约束。
- 同一批次时隙、批次内接口范围和加工输出版本均有数据库唯一约束。
- 加工队列与重试索引的部分条件和设计文档一致。
- Ruff、mypy、pytest 和 Alembic head 检查通过。

## 风险与回滚

当前处于开发阶段且没有生产数据，模型变化直接更新最新基线迁移，测试库从零重建，不保留旧数据库兼容逻辑。首次正式上线后才冻结迁移历史并使用新增迁移演进。
