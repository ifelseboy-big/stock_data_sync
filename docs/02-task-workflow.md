# 任务调度与数据加工流程

## 1. 目标

系统将数据采集和数据加工设计为两类独立任务。采集任务只负责从 Tushare 获取并保存原始数据；数据加工任务基于一个或多个接口的原始数据完成关联、清洗、校验、计算和正式入库。

数据加工不能在单个接口数据刚落地时立即触发。系统必须先等待当前采集批次结束，再统一生成加工计划，并通过唯一入口受控并发执行，避免瞬时数据库和磁盘 IO 压力。DATE 数据集的不同业务日期允许并行，同一日期保持串行；GLOBAL、MONTH 和 ENTITY 范围按输出数据集串行。

## 2. 核心对象

| 对象 | 定义 |
|---|---|
| 采集任务 | 针对一个 Tushare 接口和指定数据范围执行原始数据获取 |
| 采集批次 | 组织同一业务周期内需要执行的一组采集任务 |
| 原始数据资产 | 采集任务落地的原始数据及其接口、范围、版本和获取时间 |
| 加工计划 | 采集批次结束后，根据原始数据依赖统一生成的加工任务集合 |
| 数据加工任务 | 消费一个或多个原始数据资产，生成正式数据 |
| ApiSpec | 统一声明接口字段、采集范围、拆分、空结果和重试规则 |
| DatasetSpec | 统一声明输出数据集、处理器版本、必要依赖、写入策略和质量规则 |
| 数据发布 | 登记某一数据集范围已经完整加工并可供消费者使用 |

采集任务和数据加工任务不共享任务实例，也不能直接调用对方。两者通过原始数据资产和依赖关系连接。

## 3. 总体流程

```text
创建采集批次
    ↓
执行批次内的采集任务
    ↓
所有采集任务进入终态
    ↓
关闭采集批次并封存原始数据版本
    ↓
统一解析数据加工依赖
    ↓
生成加工计划和待执行队列
    ↓
通过统一入口受控并发执行加工任务
    ↓
事务写入正式表并登记数据发布
```

## 4. 数据采集要求

1. 每个采集任务只负责一个 Tushare 接口及其数据范围。
2. 采集任务通过统一 Provider 完成请求；限流、超时和物理请求重试不能散落在任务代码中。
3. 采集任务不得执行跨接口关联、业务清洗、聚合计算和正式数据入库。
4. 同一采集批次内的任务相互独立，可以在 Tushare 总请求额度内并发执行。
5. 采集成功的判断标准是对应原始数据已经完整、可靠地落地。
6. 采集任务完成后只更新原始数据就绪状态，不直接启动数据加工任务。
7. 采集批次必须等待批次内所有采集任务进入成功、失败、跳过或取消等终态后才能关闭。
8. 采集批次关闭时必须封存本批次使用的原始数据版本，后续到达的数据不能自动混入已生成的加工计划。

采集任务在进入失败终态前，应先完成自动重试。网络异常、数据暂未发布等可恢复错误在原批次关闭后通过新的补采批次继续处理；Token、权限、参数或接口结构变化等不可恢复错误直接进入人工处理。自动补采达到限制后仍失败的，也转为人工处理。原采集批次关闭后不得重新打开。每次领取生成唯一执行租约，原始资产写入执行代次隔离的路径，完成和失败回写必须匹配当前 token；超时回收后的旧 worker 不得登记资产或覆盖新尝试。单任务重试、批次失败任务重试和全部失败任务重试必须共用同一事务级 advisory lock；取得锁后重新检查同一接口与逻辑范围是否已有更新的活动任务或成功结果，确保并发操作只创建一个修复任务。

接口返回空数据时必须结合交易日、接口类型和任务范围判断：允许为空的记为正常空数据；按业务规则应有数据但为空的记为可恢复失败；无法判断的记为异常并告警，不能作为有效依赖参与加工。

采集计划按阶段幂等追加任务。最终阶段必须在同一事务中完成全部任务写入，并冻结 `plan_version`、`expected_task_count` 和 `planning_completed_at`。只有计划已冻结、实际任务数符合预期且所有任务进入终态，批次才能关闭，防止 Scheduler 中途退出时把未生成完整任务的批次误判为完成。

采集领取顺序固定为：数据修复、当期日常/主数据/热榜/延迟任务、历史回填。同一等级再按计划时间、接口和范围稳定排序。历史回填与日常任务共用队列和额度，但不能抢占当前生产任务；回填命令创建批次前必须自动预建覆盖日期范围内的全部月分区。

具体每日阶段、接口范围及发布时间以[Tushare 采集设计](03-tushare-collection.md)为准。APScheduler 只触发计划动作，PostgreSQL 中的批次与任务才是执行真相。

## 5. 数据加工依赖

数据加工任务可以依赖多个接口、不同范围或不同版本的原始数据。例如：

```text
加工任务 A
├── daily：指定交易日
├── adj_factor：指定交易日
└── stock_basic：截至该交易日的最新版本
```

所有声明的加工依赖均为必需依赖，不设置可选依赖。依赖既可以指向当前采集批次指定业务日期的数据，也可以指向采集批次关闭时已经存在的最新有效原始数据。

数据集依赖同时区分“执行前置条件”和“重算触发器”。例如板块成分必须依赖已发布的板块主数据和股票主数据，但主数据单独更新时不能复用覆盖范围不完整的历史成分原始包提前重算；只有新的 `ths_member` 原始批次才触发成分加工。主数据依赖仍会在成分任务执行前进行完整性校验。

依赖分为两类：

- `RAW_ASSET`：指向一个已封存原始资产；同一接口拆成多个范围时，每个范围保存独立依赖记录。
- `DATASET_RELEASE`：指向另一数据集已经发布的明确范围，例如涨跌停价补充依赖同日 `stock_daily.core`。

依赖解析结果必须持久化，不能在执行时临时查询后丢失。稳定范围键用于区分同一接口的多个日期、分片、板块或题材。

加工计划生成时必须逐项检查依赖：

- 必需依赖全部就绪后，任务进入待执行队列。
- 必需依赖缺失或采集失败时，任务进入阻塞状态并记录具体原因。
- 被阻塞任务不能使用不完整数据降级执行。
- 加工任务只能读取采集批次关闭时封存的原始数据版本。

加工规划不得在每轮调度中重扫全部历史批次。系统根据完整 DatasetSpec 目录计算稳定指纹，只领取指纹不同的关闭批次，每轮数量由 `PROCESSING_PLAN_BATCH_LIMIT` 控制、默认 100。空水位优先，同类水位内按最近关闭时间优先；同一轮已尝试但因锁竞争未完成的批次不重复占用额度。每个批次使用独立短事务；先按 `process_id` 锁定需要重新解析的非终态任务，再按完整主键顺序锁定依赖，完整解析且不存在等待活动上游发布的依赖后才更新 `processing_plan_version` 和 `processing_planned_at`。终态、运行中和等待重试的任务不参与规划锁定。升级前历史批次保持空水位并有界收敛，目录升级会使旧指纹批次分批重算；新关闭批次不会被历史积压阻塞。

采集批次部分失败时仍需生成完整加工计划。依赖完整的任务正常入队，依赖缺失的任务保持阻塞。补采成功后只解除受影响任务的阻塞，不得重新执行无关任务。

## 6. 受控并发加工要求

1. 所有采集批次、补采批次和人工重跑产生的可执行加工任务进入同一个全局加工队列。
2. 全局加工并发数由 `PROCESSING_MAX_WORKERS` 控制，默认 3、允许 1 至 8；DATE 数据集同一业务日期最多运行一个任务，不同日期可并行，其他发布范围同一输出数据集最多运行一个任务。
3. 加工任务必须按照依赖关系、业务优先级和确定性顺序领取。
4. 任一任务完成并释放执行槽位后，执行器立即从队列补位，不等待下一次 5 秒定时唤醒。
5. 某个加工任务失败时，依赖它的后续任务进入阻塞状态；与它没有依赖关系且输出数据集不冲突的任务可以继续执行。
6. 加工任务失败重试时只能重新读取已落地的原始数据，不得重新调用 Tushare。
7. 清洗或计算规则升级后，可以基于指定原始数据版本创建新的加工任务重新处理。
8. 加工任务等待重试期间不得占用执行槽位，重试时间到达后重新进入队列。

加工并发限制只针对数据加工入口。加工上一批数据时，后续采集批次仍可继续执行采集；原始数据积压或存储压力超过限制时，系统可以暂停新采集任务形成反压。

在依赖满足的前提下，加工任务优先级依次为：当期正常加工、自动补采后的加工、普通人工重跑、历史回填与规则重算。同一优先级按业务日期和进入队列时间排序，明确标记为紧急的人工任务可以提升优先级。

加工完成后，如果依赖的原始数据产生补采或修订版本，系统应创建新的加工运行，只重算受影响的数据范围，并保留原加工记录。

加工领取使用 PostgreSQL 事务级 advisory lock 串行化抢占，并使用 `FOR UPDATE SKIP LOCKED` 防止重复领取。每次领取同时检查全局运行数上限和正在运行的发布范围；进程内分发锁负责协调 5 秒定时唤醒与任务完成回调。队列按 `priority、business_date、queued_at、process_id` 稳定排序；建议数值优先级为：紧急人工 50、当期正常 100、自动修复 200、普通人工重跑 300、历史回填或规则重算 400。

所有可能同时修改加工任务和依赖的路径遵守固定加锁顺序：作用域 advisory lock（如有）→ `processing_task.process_id` 升序 → `processing_dependency` 完整主键升序 → 发布或业务表。每次加工领取生成唯一执行租约；发布和失败回写在任务行锁内必须匹配当前 token，超时恢复清空旧 token，旧 worker 不得重复发布或覆盖新尝试。批量重试先筛选逻辑任务，再只按 `process_id` 升序锁定实际候选，禁止按新旧时间反向锁住整批失败历史。

### 6.1 调度循环

| 调度动作 | 建议频率 | 职责 |
| --- | --- | --- |
| 日历维护 | 启动时、每月 | 确保本地交易日历覆盖当前年和下一年 |
| 阶段计划 | 每日阶段时刻及启动补跑 | 创建或复用批次，幂等追加任务并冻结最终计划 |
| 采集派发 | 每 5 秒兜底唤醒 | 填满空闲线程；任务完成后立即补位，不在事务内调用 Tushare |
| 批次关闭 | 每 10 秒 | 检查计划完整性及全部任务终态 |
| 加工计划 | 每 10 秒 | 为已关闭批次生成加工任务并解析必要依赖 |
| 加工派发 | 每 5 秒兜底唤醒 | 填满配置的并发槽位；任务完成后立即补位 |
| 状态协调 | 启动时、每 5 分钟 | 恢复异常运行状态、孤儿文件、分区和发布一致性 |

每个可管理调度任务都有 PostgreSQL 启停记录和执行记录。定时触发、人工触发与启动补偿执行同一个函数注册表，并按 `job_id` 共用会话级 advisory lock；唯一部分索引保证同一业务任务最多一条 `RUNNING`。人工派发器按创建顺序尝试全部待执行 Job；advisory lock 正被真实执行占用时跳过该 Job 并继续下一个。取得某个 Job 的 advisory lock 后，执行器先把该 Job 因上次结果回写失败而遗留的 `RUNNING` 收口为 `FAILED`，再创建或转换本次执行记录，因此无需等待进程重启自愈。人工派发器允许在 `SCHEDULER_MAX_WORKERS - 1` 的现有线程容量内启动多个实例，因此单个长人工任务不会在选择层或执行层造成全局队头阻塞，也不能占满 Scheduler；配置下限为 2，始终给自动采集、加工、关闭和异常恢复保留至少一个线程。Scheduler 启动并取得单例锁后仍会全量收口上次进程遗留的 `RUNNING`，再执行启动补偿；单个业务任务失败时记录失败并继续启动。已经关闭的同一调度时隙视为本周期已结束，不因代码目录变化重新打开，新增范围通过修复批次或下一周期处理。

任务领取只能在短事务内使用条件更新或 `FOR UPDATE SKIP LOCKED` 完成状态切换，接口调用、Parquet 读写和长时间计算均不得占用领取事务。

### 6.2 正式写入与发布

加工任务先把转换结果写入会话级临时表并执行质量校验，再按数据集采用以下策略：

| 策略 | 用途 |
| --- | --- |
| `UPSERT_KEY` | 有稳定自然键的行情、指标和资金数据 |
| `REPLACE_DATE` | 热榜、龙虎榜、题材成员等可能整体修订的数据 |
| `REPLACE_ENTITY` | 概念成员当前快照、指数月度成分等实体范围 |
| `MASTER_MERGE` | 股票、ETF、指数、公司和概念主数据 |
| `PATCH_COLUMNS` | 只补充已发布核心记录的指定字段 |

正式表写入、质量门禁通过和 `dataset_release` 更新必须位于同一 PostgreSQL 事务。`dataset_release` 表示某个范围的当前完整性和血缘，不代表业务表保存了可直接切换的历史版本；恢复旧结果必须从对应原始资产重新加工并发布新版本。

阻断级质量校验至少覆盖资产哈希、结构指纹、主键唯一、业务日期、代码外键、行数变化、金额和成交量换算及已声明的跨接口一致性。跨接口字段必须区分事实字段与可选增强字段：事实字段或必需依赖异常时阻断并回滚；已确认存在供应方历史覆盖缺口的可选增强字段按市场和范围隔离为空、记录质量警告并继续发布已经通过自身校验的事实数据。例如北交所历史 `daily_basic` 缺失或与 `daily` 错位时，只隔离估值字段，不能丢弃 `daily + adj_factor` 行情。沪深同类异常仍执行严格比例和数量阈值。

### 6.3 反压

加工并发只能在配置上限内调整，不能靠无限增加线程掩盖积压。当队列深度、最老等待时间、磁盘保留空间或数据库状态超过阈值时，先暂停历史回填和普通人工任务；达到存储保护线后暂停除紧急修复外的新采集。阈值必须可配置并进入告警，不能写死在页面中。

## 7. 任务状态与人工操作

采集批次至少需要表达：

```text
等待采集 → 采集中 → 批次关闭 → 已生成加工计划
```

采集任务至少需要表达：

```text
待执行 → 执行中 → 成功
                ├→ 合法空结果
                ├→ 等待重试 → 待执行
                ├→ 失败
                ├→ 跳过
                └→ 取消
```

数据加工任务至少需要表达：

```text
等待依赖 → 待执行 → 执行中 → 成功
                         ├→ 失败
                         ├→ 跳过
                         └→ 取消
```

后台必须支持查看依赖、排队顺序、阻塞原因和执行结果，并支持失败重试、人工跳过、取消以及基于原始数据重新加工。

所有人工补采、回填、重试、跳过和取消都必须经过任务服务进行合法状态迁移，提交原因和幂等键。Web 与 CLI 不能直接调用 Provider、直接更新正式表或强制把阻塞任务改为成功。

## 8. 运行统计

采集任务记录接口、请求次数、限流等待、接口耗时、返回数据量、重试次数和错误原因。

数据加工任务记录排队时间、依赖等待时间、读取数据量、异常数据量、写入数据量、各处理阶段耗时和失败原因。

后台必须能够从加工结果追溯到使用的原始数据资产、采集任务和采集批次。

## 9. 验收条件

1. 单个采集任务成功时不会立即触发数据加工。
2. 只有采集批次结束后才统一生成加工计划。
3. 一个加工任务能够声明并检查多个原始接口依赖。
4. 同一时刻加工任务数不超过配置上限；DATE 数据集不同日期可并行且同日不重叠，其他范围同一输出数据集最多运行一个任务。
5. 加工失败和重试不会再次请求 Tushare。
6. 缺少必需原始数据时，加工任务保持阻塞且不得使用不完整数据执行。
7. 加工结果能够完整追溯到实际使用的原始数据版本。
8. 加工任务等待重试时，其他无依赖且输出数据集不冲突的任务能够继续执行。
9. 加工期间允许其他采集批次继续采集，采集和加工线程池互不占用。
10. 原始数据补采或修订后，只重新加工受影响的数据范围。
11. 重复触发同一计划不会创建重复批次、采集任务或加工输出版本。
12. Scheduler 在计划生成、文件封存或正式写入期间退出后，重启协调能够恢复到一致状态。
13. 正式表写入和数据发布同事务提交，消费者不能看到半成品。
14. 队列和磁盘触发反压时，日常及紧急修复优先，历史回填不会挤占生产链路。
