# 管理后台与可观测性设计

## 定位

管理后台是数据同步系统的运维与观测入口，不承担股票行情分析。它用于回答四类问题：系统现在是否正常、任务运行到哪里、失败或阻塞发生在哪里、是否需要人工介入。

## 功能结构

### 运行中心

- 运行概览：展示正在采集的批次、当前加工任务、加工排队与阻塞数量、任务成功率和接口健康摘要。
- 采集运行：按采集批次查看任务完成情况、自动补采结果、失败数量与耗时。
- 加工队列：展示当前运行任务、受控并发队列、优先级、等待重试和阻塞原因。
- 数据就绪检查：按加工任务汇总必要的原始采集数据和已发布数据，默认只显示未全部就绪的任务；展开后按数据来源聚合数量、就绪状态和缺失原因，不平铺单个范围记录。
- 数据发布：展示各数据集的最新发布范围、发布时间、行数变化和处理器版本；支持按日期范围逐交易日检查全部日频数据集是否完整，并列出缺失项。
- 任务调度：展示全部定时任务、计划、启停状态、下次执行时间、最近结果、耗时和失败原因，并查询历史执行记录。
- 发布完整性：按截至今天的最近交易日对比全部日频数据集发布记录，明确列出缺失数据集；不得使用交易日历中的未来日期。

### 观测中心

- 接口监控：展示应用配置的 Tushare 每分钟安全预算、最近 60 秒实际请求数、主动平滑排队数，以及各接口的成功率和耗时分位数；平滑排队是本地保护行为，不表示供应方返回限流；本地预算也不是供应方实时余额。
- 运行记录：查询采集与加工任务的执行实例、重试次数、耗时及错误摘要。
- 告警中心：聚合任务失败、依赖阻塞、接口异常和系统异常；确认告警属于后续写操作能力。
- 系统运维：展示后端、数据库、调度器等组件状态和版本信息；PostgreSQL 分别展示客户端总连接、活跃连接、空闲连接、空闲事务连接和后台进程，不把 `pg_stat_activity` 全部记录误称为活跃连接；同时区分 PostgreSQL 共享缓冲区运行值与 API 进程启动后的峰值内存。启停服务由安装目录中的运维命令负责，不在浏览器中直接执行。
- 追溯详情：从发布记录依次跳转到加工任务、依赖、原始资产、采集任务和采集批次，不在列表接口中返回完整请求参数。

## 指标口径

| 指标 | 口径 |
| --- | --- |
| 采集中批次 | 当前状态为运行中，且尚未关闭的采集批次数量 |
| 加工中任务 | 当前处于运行中的加工任务数，不超过服务端配置上限；同一输出数据集最多一个 |
| 阻塞任务 | 因必要依赖未就绪而无法进入加工队列的任务数量 |
| 今日任务成功率 | 今日已进入终态的任务实例中，成功实例数 / 终态实例数；运行中、排队中和等待重试不进入分母 |
| 今日接口成功率 | 今日已完成的 Tushare 请求中，业务判定成功的请求数 / 已完成请求数；限流等待尚未发出时不进入分母 |
| 接口 P95 耗时 | 今日已完成请求从发出到收到完整响应的耗时第 95 百分位，不包含限流器中的排队等待 |
| 任务耗时 | 任务实际开始执行至进入终态的墙钟时间；队列等待和重试退避分别记录，不混入执行耗时 |
| 批次耗时 | 首个采集任务开始至所有采集任务进入终态并关闭批次的墙钟时间 |
| 当前窗口预算 | Tushare 最近 60 秒滑动窗口内已实际发出的请求数量；上限来自服务端配置，不是供应方实时余额 |
| 发布延迟 | 数据集实际发布时间减去该数据集配置的目标发布时间；尚未发布时按当前时间计算 |
| 队列深度 | 当前处于待执行状态的加工任务数量；等待依赖、阻塞和等待重试分别统计 |
| 最老排队时间 | 当前可执行加工队列中最早 `queued_at` 到采样时刻的时长 |
| PostgreSQL 客户端连接 | `pg_stat_activity.backend_type = 'client backend'` 的连接总数；后台进程单独统计 |
| PostgreSQL 活跃连接 | 客户端连接中 `state = 'active'` 的数量，不包含执行本次监控查询的连接 |
| PostgreSQL 空闲事务连接 | 客户端连接中 `state` 为 `idle in transaction` 或 `idle in transaction (aborted)` 的数量；正常稳定状态应为 0 |

所有比率在分母为零时返回 `null`，前端显示 `--`，不显示为 0%，避免把“没有样本”误解为“全部失败”。所有时间由服务端返回带时区的 ISO 8601 字符串，前端按用户时区展示。

Scheduler 使用 PostgreSQL 会话级 advisory lock 保证单实例。锁连接在 Scheduler 生命周期内保持，但获取锁的隐式事务必须立即提交；会话级锁在提交后仍有效，关闭 Scheduler 时主动解锁并再次提交，稳定状态下不得因此产生长期 `idle in transaction`。

## 管理 API

### 查询接口

| 接口 | 数据来源与内容 |
| --- | --- |
| `GET /operations/overview` | 批次、任务、加工槽位、接口额度和最近异常聚合 |
| `GET /operations/acquisition-batches` | 采集批次、任务进度、空结果、重试和资产封存状态 |
| `GET /operations/processing-queue` | 全局队列、依赖就绪数、等待重试和阻塞原因 |
| `GET /operations/dependencies` | 原始资产及已发布数据集依赖的解析结果 |
| `GET /operations/releases` | 最新发布业务日、范围、延迟、行数变化和缺失范围 |
| `GET /operations/release-coverage` | 截至今天最近交易日的日频数据集发布完整性 |
| `GET /operations/providers/tushare` | 请求量、成功率、P50/P95、重试、空结果和额度快照 |
| `GET /operations/runs` | 采集与加工执行实例、尝试次数、耗时和错误摘要 |
| `GET /operations/scheduled-jobs` | 定时任务计划、启停状态、下次执行时间和最近执行结果 |
| `GET /operations/scheduled-job-executions` | 定时、人工和启动补偿执行记录 |
| `GET /operations/alerts` | 任务、依赖、发布、额度、存储和服务异常 |
| `GET /system/resources` | 数据库连接、锁、长事务、库大小、CPU、内存、磁盘和目录容量 |

列表统一支持状态、接口、数据集、业务日期、批次和分页过滤。概览返回聚合结果，列表默认限定最近 24 小时或 30 天，禁止无时间范围扫描全部运行历史。所有响应返回 `generated_at`；Tushare Token、数据库口令、完整错误响应和未脱敏请求参数不能返回前端。

### 人工操作

目标能力包括创建历史回填、创建修复批次、按发布缺失范围精准回填或修复、重试采集任务、重试加工任务、跳过和取消，以及启用、停用和人工执行定时任务。数据发布页提供常用时间范围，并允许对选中范围的全部历史缺失或单个缺失交易日发起恢复；服务端会重新核对当前发布和活动采集任务，只为仍缺失的数据集创建必要接口任务。所有写请求必须提交 `reason` 和 `Idempotency-Key`，只允许合法状态迁移；重复请求返回首次创建的相同资源。

人工操作只调用任务 Service 并写入 PostgreSQL 队列。管理 API 不直接请求 Tushare、不等待长任务完成、不直接更新正式业务表，也不能修改已封存原始资产或强制把阻塞任务标记为成功。

单机自用阶段由服务端读取 `ADMIN_API_TOKEN`，管理页面通过 `/system/admin-config` 自动获取并作为 Bearer Token 调用写接口，不要求用户重复输入。Tushare Token 与管理 Token 仍保持分离。每次已受理操作将幂等键、操作者、request_id、目标 ID、原因、结果和客户端地址写入 `operation_command`，同时输出结构化审计日志；被拒绝的操作写结构化审计日志。以后支持多用户时再增加账号体系。

## 告警规则

| 告警 | 级别 | 触发条件 |
| --- | --- | --- |
| Scheduler 不可用 | P1 | 状态不可达或心跳超过 3 个轮询周期 |
| Token、权限或结构错误 | P1 | 任一不可恢复的供应方错误 |
| 正式数据未按时发布 | P1/P2 | 超过数据集发布目标仍无对应发布记录 |
| 加工阻塞或积压 | P2 | 在依赖观测页展示；告警中心只保留可直接处理的根因失败，避免同一采集失败重复生成大量下游告警 |
| Tushare 异常 | P2 | 10 分钟错误率超过 20%，且请求数不少于 20 |
| 额度压力 | P3 | 当前窗口持续接近配置预算 |
| 存储保护 | P1/P2 | 触及预警线或绝对保留空间 |

告警阈值由服务端配置和数据集规范提供，前端只展示结果。同接口同范围、同数据集同发布范围或同调度任务出现更晚的成功记录后，旧失败自动退出当前告警，但继续保留在运行记录中。加工恢复按数据集目录把业务范围规范化为 `GLOBAL`、`DATE`、`MONTH` 或 `ENTITY`，再用 `dataset_release (dataset_name, scope_type, scope_key)` 精确判断当前发布；没有当前发布时才检查同范围的活动加工任务。`dc_concept_cons` 从旧逐题材范围升级为按日期分页后，以“接口+业务日”的更晚成功识别旧范围失败已经恢复；该兼容分支与普通 scope 查询分开执行，保证常规查询命中恢复索引。依赖不可用形成的 `BLOCKED` 只在加工队列和依赖观测展示，不重复进入告警中心；供应方完全重复行已经被确定性去重时仅保留任务警告记录，不进入当前告警。人工跳过和取消不是当前异常。

## 日志与消费边界

API 和 Scheduler 输出 JSON 日志，统一包含 `timestamp`、`level`、`service`、`event`、`request_id`，以及可用的批次、任务、加工、接口和数据集标识。日志按大小轮转；监控统计来自运行表和状态采样，不通过扫描日志计算。

研究数据查询必须先检查对应范围的 `dataset_release`。未发布应返回“数据尚未完整”，不能把业务表空查询解释成当天没有行情。原始 Parquet 不通过研究 API 开放。

## 前后端边界与实施状态

前端按业务页面分模块，统一从 `modules/operations/api.ts` 读取运维视图，接口类型集中在 `contracts.ts`。这些接口是只读查询模型，不直接复用任务执行域对象，避免后台展示需求反向耦合任务执行实现。

当前前端与后端已经接通运行概览、任务调度、采集批次、加工队列、数据就绪检查、数据发布及完整性、接口监控、运行记录、告警和系统资源。列表由服务端限制 30 天时间窗并提供分页和过滤；实际 Tushare 请求单独写入 `provider_request_log`，不依赖扫描日志统计。

历史回填、修复采集、任务重试、跳过、取消、批次取消和调度任务启停/人工执行已经通过服务端状态机、页面自动读取的 Bearer 管理 Token、`Idempotency-Key` 和持久命令记录接入。采集批次可直接查看其任务，既可重试单个最终失败项，也可把全部尚未恢复的失败项合并为一个修复批次；加工队列单独展示尚未被后续发布替代的最终失败任务，支持单个或全部重新入队，批量操作会跳过依赖未就绪项并返回数量。调度执行队列、耗时和失败原因保存在 PostgreSQL；重复命令返回首次结果。管理 API 只写 PostgreSQL 队列，不直接调用 Tushare 或更新正式业务表。告警确认、外部通知和优先级提升尚未纳入当前写操作范围。
