# Tushare 请求策略

```text
任务定义
   ↓
StockService：决定交易日、批次和增量范围
   ↓
TushareProvider：统一限流、超时、重试和指标
   ↓
Tushare API
```

## 规则

1. Tushare 硬限制按 500 次/分钟配置，应用预算默认 480 次/分钟。
2. 使用平滑限流，每约 125ms 释放一个请求，不在分钟边界突发发送。
3. 调度器所有线程和所有 TushareProvider 实例共享同一个额度。
4. 调度器通过 PostgreSQL advisory lock 保证单实例，防止多个进程分别计数。
5. 每次网络重试都要重新申请额度；权限、参数等业务错误不重试。
6. 单次请求超时默认 30 秒，网络异常最多尝试 3 次，采用指数退避和随机抖动。
7. 指标区分物理请求耗时、逻辑查询总耗时、限流等待和物理请求成功率。
8. 手工补数只能提交为调度任务，Web API 不直接调用 Tushare。
9. 业务同步优先使用按交易日返回全市场数据的接口，避免逐股票循环请求。

当前限流器以“单调度器进程”为边界。如果未来必须部署多个实际执行节点，需要将限流状态
升级为 PostgreSQL 全局配额或专用分布式限流服务，不能简单增加调度器实例。
